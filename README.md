### Анализ и рефакторинг

1. Какие проблемы были в исходной реализации списания?

```
Первая проблема это то, что обе операции с БД выполняются в отрыве друг от друга, соответственно, между этими операциями есть окно, при коротором возможна параллельная операция с изменением данных, и это приведет к неверным данных в последующих операциях.

Вторая проблема: неверный расчет текущего баланса. В расчет включались просроченные начисления, а также не учитывались списания пользователя.

Проблема номер три: повторные или одновременные запросы с тем же самым request id будут создавать дубликаты списаний и баланс в итоге может уйти в минус.

И последнее нет корректной обработки запросы с выбросами понятных пользователю ошибок.
```

2. Какие проблемы при одновременных запросах вы устранили и как именно?

```
Чтобы правильно расчитать баланс при параллельных запросах я обернул все операции в общую транзакцию (создал scope так сказать) и заблокировал обновление нужных для расчета баланса строк. То есть теперь второй параллельный запрос будет ожидать завершения первого. И баланс не уйдет в минус.

А проблему с двойным списанием: когда повторно один и тот же пользователь сумел отправить запрос на списание с тем же request id я решил на уровне БД, создав ограичение уникальности по user_id и request_id. Дополнительно я проверяю есть ли в наличии такая запись и отправляю ответ со статусом 409 и dublicated: true.
```

3. Какие обязательные правила корректности вы обеспечили на уровне БД и на уровне кода?

```
На уровне БД я добавил ограничение уникальности по составному ключу (user_id, request_id) bonus_transactions_user_id_request_id_uq. Это будет обеспечивать идемпотентность данных.

На уровне кода делаю проверки: соответствует ли payload, валидация payload (amount > 0, наличие requestId или Idempotency-Key header); проверяю просрочку бонуса (expires_at). Также отправляю response с правильными статусами.
```

4. Как реализована защита от повторного одинакового запроса (`duplicate request`)?
```
Опять же на уровне бд уникальный индекс не позволяет создать запись для того же пользователя с тем же requestId.

На уровне логики мы сначала проверяем есть ли записи с тем же user_id и request_id. Далее проверяем соответствие payload: если не совпадает то возвращаем ответ `Dublicated request, 409`, в противном случае создаем транзакцию и возвращаем `Success: true, dublicated: true` ответ с кодом 200.

```

5. Как обеспечено, что просроченное начисление (`expired accrual`) не участвует в балансе?

```
Поскольку при начислении бонуса мы присваиваем полю expires_at дату ее просрочки (дата начисления плюс тридцать дней), мы можем проверить в момент списания если expires_at дата больше даты списания: в этом случае нам ясно, что это начисление больше не актуально и мы не можем его учитывать при расчете баланса.
```
